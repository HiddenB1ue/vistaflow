# Feature Specification: Railway Crawl Task Types

**Feature Branch**: `002-railway-crawl-tasks`  
**Created**: 2026-04-05  
**Status**: Draft  
**Input**: User description: "新增三个铁路采集任务类型：爬取车次、爬取车次经停、获取某天运行的车次。参考 other\\to-rome 的抓取方式，但核心逻辑遵循 VistaFlow 当前任务模块，仅后端。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 配置铁路采集任务 (Priority: P1)

作为后台运营人员，我可以在现有任务管理接口中创建、查看和维护三种新的铁路采集任务，这样我就能把车次、经停和开行数据采集纳入统一的任务中心，而不是依赖临时脚本。

**Why this priority**: 没有可配置的任务定义，就无法复用现有任务中心，也无法稳定触发后续数据采集。

**Independent Test**: 仅实现这部分后，管理员也应能通过任务类型目录查看三个新类型，并成功创建、更新、读取带有合法参数的任务定义。

**Acceptance Scenarios**:

1. **Given** 管理员查看任务类型目录，**When** 系统返回任务类型列表，**Then** 列表中包含“爬取车次”“爬取车次经停”“获取某天运行的车次”三个可选类型及各自参数说明。
2. **Given** 管理员提交一个“爬取车次”任务，**When** 提供日期和关键字这两个必填参数，**Then** 系统成功保存任务定义并返回规范化后的任务载荷。
3. **Given** 管理员提交一个缺少必填参数或参数格式错误的新任务，**When** 系统校验请求，**Then** 系统拒绝保存并返回明确的校验错误，而不会生成无效任务。

---

### User Story 2 - 同步车次与经停数据 (Priority: P1)

作为后台运营人员，我可以手动执行“爬取车次”和“爬取车次经停”任务，把采集结果写入现有铁路主数据表，这样规划、搜索和后续运行状态同步都能依赖最新的车次基础数据。

**Why this priority**: 车次主表和经停表是后续运行状态、票价和路径规划的基础，没有这两类任务，新任务模块对铁路数据的价值不完整。

**Independent Test**: 仅实现这部分后，管理员应能创建并执行车次任务与经停任务，且在运行完成后看到写入计数、运行状态和对应数据落库结果。

**Acceptance Scenarios**:

1. **Given** 已创建“爬取车次”任务，**When** 管理员手动触发执行，**Then** 系统采集符合关键字和日期条件的车次，并将新增或更新后的车次记录写入车次主数据。
2. **Given** 已创建“爬取车次经停”任务，**When** 管理员手动触发执行，**Then** 系统采集目标车次在指定日期的经停明细，并将经停记录写入经停数据，同时保证相关车次主数据可被关联。
3. **Given** 上述任一任务执行成功，**When** 管理员查看任务运行详情，**Then** 系统展示本次执行状态、摘要、处理数量和运行日志。

---

### User Story 3 - 同步指定日期的运行车次记录 (Priority: P2)

作为后台运营人员，我可以执行“获取某天运行的车次”任务，补齐指定日期与指定车次标识对应的开行记录，这样系统可以判断某日哪些车次可视为运行中，并为下游能力提供按日运行事实。

**Why this priority**: 该能力依赖车次基础数据，但它直接决定按日运行视图是否准确，是铁路数据链路中的关键补充能力。

**Independent Test**: 仅实现这部分后，管理员应能执行运行车次任务，并在完成后看到指定日期的开行记录已更新且可在任务运行详情中追踪结果。

**Acceptance Scenarios**:

1. **Given** 已创建“获取某天运行的车次”任务，**When** 管理员提供日期和车次标识并执行任务，**Then** 系统写入该日期对应的车次开行记录，并更新相关任务运行结果。
2. **Given** 同一日期与同一车次已存在运行记录，**When** 管理员再次执行同一任务，**Then** 系统不会制造重复的逻辑记录，并会按既定策略更新已有运行事实。
3. **Given** 采集源未返回任何可用运行记录，**When** 任务结束，**Then** 任务运行结果应标记为失败或空结果，并向管理员提供可操作的原因说明。

## Edge Cases

- 管理员创建任务时提供空白关键字、空白车次标识、非法日期或不支持的任务类型，系统必须在保存前拒绝请求。
- “爬取车次”任务查询结果为空时，任务必须明确区分“执行成功但无数据”与“采集失败”，避免误判为已同步。
- “爬取车次经停”任务在无法定位目标车次主记录或采集不到经停数据时，必须返回明确错误，不得写入不完整经停数据。
- “获取某天运行的车次”任务重复执行时，系统必须保持运行事实唯一且可重复同步，不得产生重复逻辑记录。
- 上游采集源超时、返回结构异常或临时不可用时，任务运行必须记录错误详情，并保持任务定义可继续重试。
- 管理员在已有任务运行中的情况下再次触发同一任务，系统必须阻止并提示该任务正在执行中。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 在任务类型目录中新增三个可创建的后端任务类型，分别覆盖车次采集、车次经停采集和按日运行车次采集。
- **FR-002**: 系统 MUST 为“爬取车次”任务要求日期与关键字两个必填参数，并在创建、更新与执行前校验参数完整性与格式。
- **FR-003**: 系统 MUST 为“爬取车次经停”任务要求日期与车次标识两个必填参数，并在创建、更新与执行前校验参数完整性与格式。
- **FR-004**: 系统 MUST 为“获取某天运行的车次”任务要求日期与车次标识两个必填参数，并在创建、更新与执行前校验参数完整性与格式。
- **FR-005**: 系统 MUST 允许管理员通过现有任务管理接口创建、读取、更新、列出和手动执行这三类任务，而不引入独立的专用管理入口。
- **FR-006**: 系统 MUST 在执行“爬取车次”任务时，将采集结果写入现有车次主数据，并确保重复执行不会产生重复逻辑车次。
- **FR-007**: 系统 MUST 在执行“爬取车次经停”任务时，将采集结果写入现有车次经停数据，并确保写入的经停记录能够关联到对应车次主数据。
- **FR-008**: 系统 MUST 在执行“获取某天运行的车次”任务时，将采集结果写入现有按日运行数据，并确保同一车次同一日期只有唯一的运行事实。
- **FR-009**: 系统 MUST 在每次任务运行中记录开始时间、结束时间、运行状态、处理数量、摘要和日志，以便管理员审计执行过程。
- **FR-010**: 系统 MUST 在采集结果为空、采集源报错、参数非法或目标车次无法解析时，向任务运行记录写入可操作的错误或空结果说明。
- **FR-011**: 系统 MUST 阻止同一任务在已有活动运行时被重复触发，并向调用方返回冲突提示。
- **FR-012**: 系统 MUST 保持现有已实现任务类型的行为不变，新任务类型的引入不得破坏现有任务查询、创建、运行和终止能力。
- **FR-013**: 系统 MUST 让管理员通过现有任务运行查询接口看到三类新任务的最近运行结果，包括成功、失败、终止和空结果对应的摘要信息。
- **FR-014**: 系统 MUST 在接入这三类新任务时保持现有 12306 采集调用协议不变，包括请求参数含义、调用方式与响应结构解释口径不变；调整范围仅限 VistaFlow 内部的任务编排、校验、日志与落库流程。

### Key Entities *(include if feature involves data)*

- **Railway Crawl Task Definition**: 管理员在任务中心配置的采集任务，包含任务类型、启用状态、手动触发能力和任务参数载荷。
- **Train Record**: 车次主数据，表示某个可识别车次及其基础信息，是经停和运行事实的关联主体。
- **Train Stop Record**: 车次在完整线路中的经停明细，包含站序、站名和到发时间等业务信息。
- **Train Run Record**: 某个车次在特定日期的运行事实，用于表达该车次在该日是否处于开行状态。
- **Task Run Summary**: 每次任务执行后形成的结果摘要，包含执行状态、处理数量、错误原因与日志线索。

## Backend Impact *(mandatory)*

### API Contracts

- **Endpoints**: 继续使用现有任务管理接口；`GET /task-types` 需要返回三个新任务类型及其参数说明，`POST /tasks` 与 `PATCH /tasks/{task_id}` 需要接受并校验三类新任务的参数载荷，`POST /tasks/{task_id}/run`、`GET /tasks/{task_id}/runs`、`GET /task-runs/{run_id}`、`GET /task-runs/{run_id}/logs` 需要正确反映新任务类型的执行结果。
- **Error Model**: 对缺失参数、参数格式错误、不支持的任务类型、任务执行冲突、车次无法解析、采集无结果和外部采集失败返回一致的业务错误，并在任务运行日志中保留可追踪说明。
- **Auth/Security**: 仍沿用现有管理员鉴权模型；新任务不引入新的公开入口、额外身份角色或新的敏感凭据暴露面。

### Data & Persistence

- **Storage Impact**: 任务定义与任务运行继续落在现有任务相关存储；三类新任务分别写入现有车次主数据、经停数据和按日运行数据，并需要保持这些业务记录的唯一性与可重复同步能力。
- **Compatibility**: 对现有任务管理接口保持向后兼容；新增任务类型后，旧客户端若不使用新类型仍可按原方式工作，已存在的任务定义与历史运行记录不需要人工迁移。

### Integrations & Operations

- **External Services**: 需要复用现有铁路采集来源，并允许参考 `other/to-rome` 的抓取方式来获取车次、经停和按日运行数据；其中 12306 侧的请求参数、调用方法和响应结构保持不变，仅调整 VistaFlow 内部承载方式；当上游不可用时，任务运行需安全失败并允许后续重试。
- **Observability**: 新任务必须产生日志、执行摘要、处理数量和最近运行状态，保证管理员可以从任务列表、运行详情和日志接口追踪采集成败与数据同步规模。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 管理员能够仅通过现有任务管理接口成功创建三类新任务，且 100% 的非法参数请求会在任务保存或执行前被拒绝。
- **SC-002**: 对于采集源返回有效数据的手动执行场景，95% 的任务运行都能在一次执行内完成数据落库，并在运行详情中显示处理数量与摘要。
- **SC-003**: 对同一业务对象重复执行相同任务后，不会产生重复的逻辑车次、重复的经停站序记录或重复的同日运行事实。
- **SC-004**: 当任务因无数据、解析失败或上游异常而未成功完成时，管理员在运行详情或日志中都能在 1 分钟内定位到明确失败原因。

## Assumptions

- 该需求仅覆盖后端任务目录、任务执行和数据落库能力，不包含新的前端页面或新的手工导入界面。
- 三类新任务继续复用现有管理员鉴权、任务中心接口和手动触发模式；自动调度能力如存在，仍遵循现有任务模块规则。
- “获取某天运行的车次”默认按 `日期 + 车次标识` 采集并写入对应运行事实，这一参数模式与 `to-rome` 中同类任务保持一致。
- 现有铁路业务表仍是系统唯一事实来源，新任务以增量同步和重复执行安全为目标，而不是替换现有数据模型。
- 可以参考 `other/to-rome` 的抓取方式和数据归一化经验，但 VistaFlow 仍需通过自身任务运行、日志和落库流程来承载该能力。
- 12306 集成协议本身不在本次变更范围内，默认继续沿用现有参数命名、请求组织方式和响应解析结构。
