# Feature Specification: Task Module Refactor

**Feature Branch**: `003-task-module-refactor`  
**Created**: 2026-04-07  
**Status**: Draft  
**Input**: User description: "重构 VistaFlow 后端任务模块，使任务类型定义自描述化、注册集中化、handler 模块化、执行上下文通用化；保持现有任务 API、数据库模型与 fetch-station、fetch-trains、fetch-train-stops、fetch-train-runs 行为兼容，降低新增任务类型时需要修改的中心文件数量，并为未来新增 price、geocode 及非铁路任务提供更好的扩展结构。仅后端。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 用统一方式定义和注册任务类型 (Priority: P1)

作为后端开发者，我可以通过统一的任务类型定义入口声明任务元数据、参数校验规则和执行行为，这样在新增一个任务类型时不需要同时修改多个分散的中心文件，也不会遗漏注册步骤。

**Why this priority**: 这是本次重构的核心价值。如果任务定义仍然分散在多个文件中，后续新增任务类型的维护成本和出错率都不会下降。

**Independent Test**: 在只实现这一部分后，开发者应能以统一方式接入一个新的示例任务类型或迁移一个现有任务类型，并通过任务类型目录、任务创建校验和执行入口验证注册链路完整可用。

**Acceptance Scenarios**:

1. **Given** 开发者新增一个符合规范的任务类型定义，**When** 系统启动并返回任务类型目录，**Then** 新任务类型会自动出现在统一目录中，并带有完整的标签、说明和参数定义。
2. **Given** 管理员提交某个任务类型的配置参数，**When** 系统进行创建或更新校验，**Then** 系统会基于该任务类型自身声明的参数规则完成规范化与校验，而不是依赖分散的特判逻辑。
3. **Given** 某个任务类型缺少必要的执行声明或参数声明，**When** 系统加载任务定义，**Then** 系统会阻止其作为可执行任务对外暴露，并给出明确的诊断信息或测试失败信号。

---

### User Story 2 - 在重构后继续稳定运行现有任务 (Priority: P1)

作为后台管理员，我希望现有任务中心在重构后继续使用原有接口管理和执行已实现任务类型，这样我可以在不学习新入口、不迁移历史数据的前提下继续使用任务列表、运行记录、日志和终止能力。

**Why this priority**: 任务模块重构不能以牺牲当前生产能力为代价；兼容性是这次重构是否可落地的前提。

**Independent Test**: 在只实现这一部分后，管理员应能继续对现有任务类型执行查询、创建、运行、查看运行记录、查看日志和终止运行，而接口路径、主要响应结构和历史运行数据保持兼容。

**Acceptance Scenarios**:

1. **Given** 系统中已经存在 `fetch-station`、`fetch-trains`、`fetch-train-stops`、`fetch-train-runs` 任务定义，**When** 管理员继续使用现有任务管理接口，**Then** 这些任务仍可被正常读取、更新、运行和终止。
2. **Given** 管理员查看历史运行记录和运行日志，**When** 任务模块完成重构，**Then** 现有运行状态、摘要、日志和进度快照语义保持一致，不需要人工迁移才能读取。
3. **Given** 某个现有任务执行成功、失败或被终止，**When** 管理员查看任务列表和运行详情，**Then** 系统仍会显示一致的最近运行状态、错误信息、摘要和处理结果。

---

### User Story 3 - 为未来不同风格的任务预留一致扩展点 (Priority: P2)

作为后端开发者，我希望未来新增同步类、补全类、查询类或其他非铁路任务时，都能复用统一的生命周期、日志、进度和依赖注入约定，这样任务模块不会随着任务数量增长而越来越杂乱。

**Why this priority**: 当前任务主要集中在铁路采集，但后续 `price`、`geocode` 或其他非铁路任务会带来不同风格的执行逻辑；如果不提前统一扩展点，模块会再次碎片化。

**Independent Test**: 在只实现这一部分后，开发者应能为一个非铁路或非采集风格的示例任务定义依赖、日志和进度约定，并证明它不需要继续膨胀已有中心上下文或复制粘贴既有 handler 模式。

**Acceptance Scenarios**:

1. **Given** 开发者新增一个与铁路采集不同风格的任务类型，**When** 它声明自身依赖、执行行为和生命周期元数据，**Then** 系统仍能以统一方式调度、记录日志和写入运行结果。
2. **Given** 任务类型需要不同的执行上下文或进度细节，**When** 它接入统一框架，**Then** 系统允许任务类型扩展自己的细节而不会破坏公共运行协议。
3. **Given** 项目中任务类型数量继续增长，**When** 维护者阅读任务模块结构，**Then** 能清晰区分通用框架能力与具体任务实现文件，而不会在单个中心文件中查找所有定义和逻辑。

## Edge Cases

- 某个任务类型只声明了元数据但未声明参数规则或执行行为时，系统必须阻止其进入可执行目录，避免形成半注册状态。
- 某个任务类型只支持手动执行或不支持终止时，系统必须通过统一的能力声明表达限制，而不是依赖隐式判断。
- 现有任务类型迁移到新注册结构时，系统必须保证原有任务定义中的 `type`、payload 和历史运行记录仍可被识别。
- 任务模块重构后，如果运行时无法加载某个任务类型定义，系统必须给出明确错误，不得静默降级为未知任务。
- 不同任务类型对进度快照的细节字段各不相同，但系统必须继续保证通用字段结构稳定，便于列表页和运行详情消费。
- 新增未来任务类型时，若未显式声明日志、摘要或结果指标约定，系统必须有一致的默认行为，避免任务运行结果不可观测。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 提供统一的任务类型注册机制，让每个任务类型能够在一个聚合定义中声明自身标识、显示名称、描述、执行能力和参数规范。
- **FR-002**: 系统 MUST 让任务类型自身声明其参数校验与规范化规则，并在任务创建、更新和执行前复用同一套规则。
- **FR-003**: 系统 MUST 让任务类型自身声明其执行处理入口，使系统不需要在多个中心文件中重复维护任务类型到执行逻辑的映射。
- **FR-004**: 系统 MUST 将具体任务实现与通用任务框架能力分离，保证维护者可以清晰区分框架层代码与任务类型代码。
- **FR-005**: 系统 MUST 保持现有任务管理接口对外兼容，包括任务类型目录、任务 CRUD、手动触发、运行查询、日志查询和终止执行的既有入口与主要响应语义。
- **FR-006**: 系统 MUST 保持现有已实现任务类型 `fetch-station`、`fetch-trains`、`fetch-train-stops`、`fetch-train-runs` 的既有业务行为、参数语义和运行状态语义不变。
- **FR-007**: 系统 MUST 允许现有任务定义、任务运行记录、任务运行日志和进度快照在不人工迁移历史数据的情况下继续被读取和使用。
- **FR-008**: 系统 MUST 提供统一且稳定的任务执行上下文约定，使任务类型可以声明自身所需依赖，同时避免为单一业务场景持续膨胀公共上下文。
- **FR-009**: 系统 MUST 为所有任务类型保留统一的生命周期协议，包括可执行性判断、开始执行、完成、失败、终止、摘要、日志和结果指标写入。
- **FR-010**: 系统 MUST 允许任务类型扩展各自的进度快照细节，但必须保持公共进度结构稳定，以兼容现有任务列表和运行详情消费方式。
- **FR-011**: 系统 MUST 为任务类型声明执行能力边界，例如是否可执行、是否支持终止、是否支持定时调度或其他能力，以支持未来新增不同风格任务而不引入隐式约束。
- **FR-012**: 系统 MUST 让新增任务类型的标准接入流程显著减少对中心文件的修改数量，并形成可测试的接入约束。
- **FR-013**: 系统 MUST 在任务类型注册缺失、定义不完整或运行时无法加载时，向开发者或管理员提供明确、可诊断的错误反馈。
- **FR-014**: 系统 MUST 为本次重构补充自动化验证，覆盖现有任务兼容性、统一注册链路、参数校验复用和新增任务接入约束。
- **FR-015**: 系统 MUST 保持现有任务模块对管理员鉴权、后台运行可见性和日志审计的既有安全与可观测要求不变。

### Key Entities *(include if feature involves data)*

- **Task Type Definition**: 单个任务类型的统一定义，包含任务标识、展示信息、能力声明、参数规则和执行入口声明。
- **Task Capability Contract**: 用于描述某个任务类型是否可执行、是否支持终止、是否支持调度及其默认生命周期约束的声明对象。
- **Task Parameter Contract**: 某个任务类型对配置载荷和运行载荷的字段要求、规范化规则和校验语义。
- **Task Execution Context Contract**: 通用任务运行上下文与任务类型专属依赖之间的边界约定，用于避免公共上下文持续膨胀。
- **Task Runtime Outcome**: 每次任务运行产生的统一结果协议，包含状态、摘要、指标、错误、日志和进度快照。

## Backend Impact *(mandatory)*

### API Contracts

- **Endpoints**: 继续使用现有任务管理 API；本次重构不新增新的对外任务管理入口，但需要保持 `GET /task-types`、`/tasks`、`/task-runs` 等接口对已实现任务类型的兼容行为。
- **Error Model**: 需要新增或明确任务类型注册缺失、定义不完整、执行能力不满足等框架级错误语义；同时保持现有业务错误和运行冲突错误的对外兼容性。
- **Auth/Security**: 继续沿用现有管理员鉴权模型；本次重构不引入新的公开入口、不改变既有权限边界，也不新增硬编码敏感信息来源。

### Data & Persistence

- **Storage Impact**: 现有 `task`、`task_run`、`task_run_log` 以及相关业务表继续作为系统记录来源；如果重构需要补充任务能力元数据或注册约束，必须评估是否需要新的存储字段，并确保历史数据兼容。
- **Compatibility**: 现有任务定义、任务运行记录、日志和进度快照必须保持向后兼容；除非设计文档显式说明并提供回滚方案，否则不得要求人工迁移现有任务数据。

### Integrations & Operations

- **External Services**: 本次重构不改变现有 12306、地图或其他外部集成的请求协议，但需要确保新的任务框架能继续承载这些集成，并为未来不同类型任务预留一致接入方式。
- **Observability**: 必须保持并明确任务列表状态、运行摘要、运行日志、错误信息和进度快照等可观测输出；重构后还应让开发者更容易判断某个任务类型是否完成注册、具备哪些能力以及为何加载失败。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 迁移后的现有四种已实现任务类型都能通过现有任务管理接口完成查询、创建、执行、查看运行记录和日志，且回归验证通过率达到 100%。
- **SC-002**: 新增一个标准任务类型时，需要修改的任务模块中心文件数量减少到不超过 2 个，并且不需要同时修改任务类型目录、参数校验映射和执行映射这三类分散入口。
- **SC-003**: 维护者能够在 30 分钟内根据文档和现有示例完成一个新任务类型的接入，并通过自动化测试验证其已正确注册和可执行。
- **SC-004**: 对于定义不完整或注册错误的任务类型，开发者能够在一次本地验证或测试运行内定位到明确失败原因，而不是在运行期通过隐式错误排查。
- **SC-005**: 重构完成后，任务模块的通用框架代码与具体任务实现代码边界清晰，新增至少一种不同风格任务时不需要继续扩展公共上下文的业务专属字段。

## Assumptions

- 本次需求仅覆盖后端任务模块及其相关设计文档、测试和必要的数据兼容处理，不包含前端页面重构。
- 现有 PostgreSQL 任务相关表和历史运行数据继续作为系统事实来源，除非设计阶段证明确有必要，否则不引入破坏性迁移。
- 现有任务管理 API 的对外路径和主要响应结构视为兼容性边界，重构应优先在内部结构上完成解耦。
- `price`、`geocode` 和其他未来任务类型作为扩展能力示例，用于约束框架设计方向，但不要求在本次重构中全部实现为正式任务。 
- 当前单进程后台执行模式可以暂时保留；本次重构重点是任务定义与执行框架的可扩展性，而不是引入分布式调度系统。
